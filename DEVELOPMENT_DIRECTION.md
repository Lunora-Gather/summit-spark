# 山巅微光开发方向守则

记录时间：2026-06-03  
用途：约束后续大步推进，避免训练、挑战、档案、UI 和发布流程互相冲突。

## 1. 分支与 PR

- 大步改造走 `codex/` 分支和 PR，不直接在本地 `main` 上继续堆。
- PR 用来集中审查一批完整体验改造：代码、文档、检查脚本、浏览器验证和发布说明必须在同一条线上。
- 不强推 `main`。如果本地和远端只有 SHA/换行差异，优先从 `origin/main` 开新分支。
- 发布前必须通过 `npm run check`、`git diff --check`、本地浏览器 smoke 和线上 marker 检查。

## 2. 产品方向

- 当前主线仍是 10 房垂直切片，不急着扩房间数量。
- P0 是开局可信、提示清楚、设置不乱、重开低摩擦。
- P1 是手感、难度类型、路线读法和训练闭环。
- P2 是长期目标：章节完成度、挑战层、长期档案、终点后的下一行动。
- 学习成熟平台跳跃的产品经验，但不复刻关卡、角色、美术、叙事或专属表达。

## 3. 数据规则

- PB、Clean、S、Style、Expert 和 Focus 仍是 mastery 的真实来源。
- 长期档案只能汇总既有真实数据，不能另开一套成绩规则。
- 控制辅助可以降低操作负担，但不能改变 PB、Clean、mastery 和挑战判定。
- Reset Focus/Drill 只清训练统计，不应该暗中改地图、PB 或设置。

## 4. UI 优先级

- 跑动中优先显示角色、危险物、资源、路线目标和死亡原因。
- 设置面板是训练驾驶舱：下一步、章节完成度、训练计划、挑战和档案必须形成一条可点击工作流。
- 终点复盘先给下一行动，再给数据详情。
- 新提示必须进入互斥规则，不能和入场卡、死亡 Coach、路线卡同时抢屏幕。

## 5. 质量门

- 新 DOM id 必须进入 `tools/check-contracts.js`。
- 新关卡数据必须进入 `tools/check-maps.js` 的规则范围。
- 新长期目标必须能从现有数据计算，不能依赖不可验证的隐式状态。
- 新 UI 必须检查桌面、移动、短视口、设置面板和终点复盘。
- 资源版本必须随公开页面功能更新。
